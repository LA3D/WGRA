The Web Forager llm-wiki collaboration story

A concrete example of how a persistent, LLM-maintained wiki turned two collaborators' routine session-work into a project-wide bug fix that neither would have caught alone.

The project in one paragraph

Web Forager is an adaptive web scraper that builds knowledge incrementally by reading webpages with Claude vision, answering user queries against what it knows, and going back to the source when it cannot. It is validated against a set of competency questions on manufacturer websites (ITAMCO, Schafer Industries, later a 108-company drone-industrial-base fleet). Two Claude Code instances contribute: one owned by Chris Sweet, one owned by Chris Frederick (Omegaice in the wiki commit attributions). Both write into a shared llm-wiki at wiki/web_forager.wiki/.

Repository: github.com/chrissweet/web_forager · Wiki: github.com/chrissweet/web_forager/wiki

The story

Chris Frederick was prepping for a fleet-scale run. The project had passed 20 of 20 competency questions on single entities (ITAMCO, Schafer). Next: scale to 108 companies. His session on 2026-06-11 audited the extraction pipeline against the platform's vision documentation and filed four new wiki pages recording what he found.

One of those pages was routine housekeeping. Vision-Token-Economics documented per-model tokenization limits, tile geometries, cost math. The kind of reference sheet you write for yourself.

A second page filed the same session was a bug report. Screenshot-Resolution-Bug recorded that browser.py was chunking screenshots only above 7680 px (the API's hard-reject ceiling), but Sonnet's native cap forces silent downscaling on anything over ~950 px tall at 1280 wide. The example: the ITAMCO homepage (1280 × 7396 px) was being read at 21% scale, 560 tokens. At 21% scale, 16 px body text renders at 3.4 px. The model had been reading thumbnails.

Chris Frederick filed the bug page with typed cross-references: criticizes: [[Experiment-11-Productionization]] in the frontmatter, related: [[Vision-Token-Economics]].

Chris Sweet reviewed the wiki entries on the other side. The cross-references made the implications visible in a way the individual finding did not. The criticizes: edge said explicitly that this bug undermined a prior experiment's conclusions. Following the link surfaced that Experiment 11 had attributed low coverage to "question/target mismatch and the plateau bug." But if pages were being read at 21% scale, Sonnet's honest "not visible on this page" answers were literally true — the pages were unreadable. Question/target mismatch remained real, but its measured magnitude had been inflated by unreadable pages.

The same review surfaced two more downstream implications also captured in the page:

  1. Model head-to-heads were confounded. The Haiku/Sonnet comparisons in Experiment 11 conflated model reasoning with what each tier could see after downscale: a 2703 px page renders at 58% on Sonnet but 95% on Opus-tier. Any Opus-vs-Sonnet decision needed post-fix data.
  2. Cost baselines were wrong by a factor of ~20. Past runs paid thumbnail prices. Fixing legibility raises image tokens ~20× on previously crushed pages (the 7396 px homepage goes from 560 illegible tokens to ~12,000 legible ones across 8 tiles).

The fix landed the same day (main-repo commit 352fe477): a pure helper chunk_scroll_positions chunks whenever the page exceeds one viewport, steps by viewport − overlap, clamps the final tile to the page bottom, caps at 20 chunks against infinite-scroll pages. Settings.viewport_height, the redundant representation of the viewport that was the source of a smaller 24 px gap defect, was deleted so the two values could not drift again.

Why the wiki structure made this catch possible

Four llm-wiki primitives did the load-bearing work.

1. Typed pages with typed edges

Every page carries YAML frontmatter declaring what kind of page it is and what its relationships are:

---
type: synthesis
up: "[[Extraction-Pipeline]]"
tags: [bug, vision, screenshots, coverage]
criticizes: "[[Experiment-11-Productionization]]"
related: "[[Vision-Token-Economics]]"
---

criticizes: is a semantic edge. It does not mean "these pages are related." It means "when you read this page, treat the target's claims as being called into question." A reviewer following that edge encounters not just the bug report but its implication for prior experiment conclusions. The edge is what turned a routine finding into a project-wide correction signal.

2. Wikilinks that make cross-page implications inspectable

The bug page names [[Experiment-11-Productionization]], [[Vision-Token-Economics]], [[Blackbox-Eval-Adoption]], [[Fleet-Run-Bottlenecks]] in its body. Each is a durable link into a page that carries its own frontmatter and its own edges. Traversing them reveals the full picture: what the bug confounded, what the fix unblocks, what runs became possible after.

3. Auto-maintained index

index_web_forager.md catalogs every page with a one-line description. A collaborator does not need to remember what exists to find it. When Chris Frederick filed four new pages in one session, the index reflected them without either of us doing extra work — the ingest procedure requires updating the index as part of the operation.

4. Per-entry commit log with author attribution

log_web_forager.md records each wiki operation as a dated entry with the first bullet reading by: <human> via <agent>. The bug-finding entry reads by: Omegaice via claude-code. The re-baseline follow-up reads the same. The mantra-clarification entries elsewhere in the log read by: Chris Sweet via claude-code. Provenance is visible on the page, and it is verifiable in git. Every log entry is its own commit (schema rule), so git blame on the log file is a faithful record of who filed what and when.

5. Session boundary crossed via durable state

Neither Claude Code session shared conversational context with the other. They shared a wiki. Chris Frederick's Claude wrote the bug page as part of ingesting session findings. Chris Sweet's Claude, in a completely separate session, read the same pages on load per the wiki-first project convention. Neither session needed to be "held open" or "handed off." The wiki was the substrate, and it survived session boundaries.

The generalizable pattern

  • Individual sessions surface findings. A developer working on prep tasks encounters ambient facts that seem too small to flag ("current chunking threshold is 7680 px, ceiling for API is 8000 px").
  • A wiki-first convention forces those findings into durable, cross-referenced structure. Not a notepad, not a Slack message, not chat context that expires. A page with frontmatter, edges, and an index entry.
  • A collaborator (or the same author in a later session) reviews the wiki structure independently. Cross-references make consequences visible that a single session cannot see, because a single session does not have both sides of the relationship in context simultaneously.
  • The typed edges tell the reviewer what kind of thinking each link warrants. criticizes: warrants doubt in the target; supports: warrants aggregation of evidence; extends: warrants inheritance of assumptions.
  • The fix cascades. Correcting the bug corrects downstream conclusions the bug confounded, and unblocks work the bug was silently gating.

None of this depends on both collaborators being in the same room, same session, or same time zone. It depends on both writing into a shared, structured, LLM-maintained knowledge base.

  5. Eval: Validity Audit — Chris Frederick's follow-up zero-API audit of the re-baseline. Shows the collaboration continuing: bug → fix → re-baseline → audit-of-re-baseline. Every hop filed in the wiki.
  6. Eval: Thinking Was the Default All Along — a second bug the same reviewer pass surfaced: adaptive thinking silently enabled by SDK default in production and every eval to date. Config made tri-state (e2b9c91f), then measured on Sonnet: adaptive 93.1% vs disabled 88.1% (P ≈ 96%). Same collaboration pattern, second finding.
  7. Blackbox-Eval Adoption — the plan the bug fix unblocked. Shows how the wiki structure sequences work: prereqs → fix → framework adoption → re-baseline.
  8. Fleet Run Bottlenecks — the prerequisite audit for the 108-company sweep. Records the seven-step prep sequence the fix was part of.
  9. Wiki Schema — Web Forager — the living document defining the frontmatter conventions, typed edges, per-entry commit log rule, and the four operations. The shape of the discipline that made the story possible.
  10. Index and Log — catalog and chronological record. Show how discoverability works: index for topic-first lookup, log for time-first lookup with author attribution.

The takeaway

  • A shared llm-wiki turns discrete sessions into cumulative project memory. Neither collaborator held state across the boundary. The wiki did.
  • Structure matters more than content volume. Frontmatter, typed edges, and cross-references made ambient session notes reviewable in a way freeform prose or chat logs are not.
  • The wiki catches bugs their filers did not consider bugs. Chris Frederick did not file Vision-Token-Economics as a bug report. Cross-referencing it against the extraction pipeline turned a routine reference page into the corner piece of a project-wide correction.
  • Attribution scales trust. by: <human> via <agent> on every log entry, one commit per entry, git-blame-traceable. Collaborators can verify who found what without asking anyone.
  • The pattern is generalizable beyond this project. Any multi-session, multi-contributor codebase where session-context evaporates can adopt the same primitives: typed pages, typed edges, index, log, per-entry commits. The llm-wiki template packages that.
